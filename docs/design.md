## Context

tinydb 是一个从零构建的 Python 嵌入式关系型数据库，目标是用于学习数据库原理并作为可嵌入的 Python 库使用。项目零外部依赖，单文件持久化。

当前状态：v0.2 已实现多表 JOIN、并发控制、CLI 增强。
约束：纯 Python 实现，零外部依赖，代码需简洁可读（教学目的）。

## Goals / Non-Goals

**Goals:**
- 提供完整的 SQL 接口（DDL/DML）
- 实现页式存储引擎 + 缓冲池
- 实现 B-tree 索引加速查询
- 实现基于 WAL 的 ACID 事务
- 提供交互式 CLI/REPL
- 支持多表 JOIN 查询（INNER/LEFT/CROSS JOIN）
- 支持多线程并发安全（读写锁）
- CLI 支持 readline、语法高亮、EXPLAIN 执行计划

**Non-Goals:**
- RIGHT JOIN / FULL OUTER JOIN
- 视图、触发器、外键、网络模式
- 行级锁、MVCC 事务隔离
- 高性能优化（以可读性优先）

## Decisions

### D1: 事务实现 — WAL（Write-Ahead Logging）

**决策**: 使用 WAL 而非影子分页。

| 方案 | 优点 | 缺点 |
|------|------|------|
| WAL | 实现简洁，崩溃恢复直观（重放日志），适合教学 | 日志文件会增长 |
| 影子分页 | 恢复速度快 | 实现复杂，需要页面级拷贝 |

**理由**: WAL 实现更简洁，日志重放的崩溃恢复逻辑易于理解和教学。

### D2: 页大小 — 4KB

**决策**: 默认页大小 4096 字节。

**理由**: 与操作系统页大小一致，是数据库系统的通用默认值（SQLite、PostgreSQL 均使用 4KB）。

### D3: SQL 方言 — 类 SQLite

**决策**: 语法尽量兼容 SQLite。

**理由**: SQLite 是最广泛使用的嵌入式数据库，用户熟悉度高，文档丰富。

### D4: 数据类型 — 静态类型 + 灵活转换

**决策**: 支持 INT、FLOAT、TEXT、BOOL 四种基本类型，插入时进行类型检查。

**理由**: 覆盖绝大多数使用场景，类型安全避免数据不一致。

### D5: 架构分层

**决策**: 采用经典数据库分层架构：

```
┌─────────────────────────────────────────┐
│              CLI / REPL                  │
│    (readline + 语法高亮 + .explain)       │
├─────────────────────────────────────────┤
│           SQL Interface                  │
│         (db.execute)                     │
├─────────────────────────────────────────┤
│  SQL Parser  │  Query Executor          │
│  (lexer/     │  (scan/join/sort/        │
│   parser)    │   aggregate/explain)     │
├──────────────┼──────────────────────────┤
│  B-tree Index│  Type System             │
├──────────────┼──────────────────────────┤
│  Concurrency │  Connection Pool         │
│  (ReadWriteLock)                         │
├──────────────┴──────────────────────────┤
│         Storage Engine                   │
│  (Page Manager + Buffer Pool + File I/O) │
├─────────────────────────────────────────┤
│       Transaction Manager (WAL)          │
└─────────────────────────────────────────┘
```

### D6: JOIN 实现 — 嵌套循环连接（v0.2）

**决策**: 使用嵌套循环连接算法，将右表物化后逐行与左表匹配。

| 方案 | 优点 | 缺点 |
|------|------|------|
| 嵌套循环 | 实现简单，正确性高 | O(n*m) 性能 |
| 哈希连接 | O(n+m) 性能 | 实现复杂，需要哈希表 |
| 排序合并 | 适合已排序数据 | 需要额外排序开销 |

**理由**: 教学数据库数据量小，嵌套循环实现简洁易懂。输出列名使用 `table.column` 限定格式避免冲突。

### D7: 并发控制 — 读写锁 + 表级粒度（v0.2）

**决策**: 实现 ReadWriteLock，以表为粒度管理锁。

| 锁策略 | 操作 |
|--------|------|
| 读锁（共享） | SELECT |
| 写锁（排他） | INSERT/UPDATE/DELETE |
| 全局写锁 | DDL（CREATE/DROP）、事务控制 |

**理由**: 表级锁粒度适中，实现简单。比数据库级锁并发度高，比行级锁实现简单。BufferPool 和 Catalog 使用 threading.Lock/RLock 保证线程安全。

### D8: CLI 增强 — 纯标准库（v0.2）

**决策**: 使用 Python 标准库 readline 实现行编辑，ANSI 转义码实现语法高亮。

**理由**: 零外部依赖约束。readline 提供行编辑和历史，ANSI 码提供关键字/字符串/数字高亮。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 纯 Python 性能较差 | 以教学可读性为首要目标，性能次之 |
| WAL 文件持续增长 | 实现 checkpoint 机制截断已提交事务的日志 |
| SQL 解析器覆盖不全 | 先支持核心语法，逐步扩展 |
| B-tree 实现复杂度高 | 从简单插入/查找开始，逐步支持删除和范围查询 |
| 嵌套循环 JOIN 性能 O(n*m) | 教学数据库数据量小可接受，后续可加索引优化 |
| 表级锁并发度有限 | 比数据库级锁好，行级锁留待后续版本 |
| readline 在 Windows 不兼容 | 主要目标平台 Linux/macOS |

## Migration Plan

不适用（全新项目，无迁移需求）。
